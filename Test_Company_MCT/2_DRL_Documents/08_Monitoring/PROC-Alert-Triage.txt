Alert Triage Procedure

Document ID: PROC-AlertTriage Version: 2.5 Effective date: February 1, 2026 Last reviewed: January 28, 2026 Next review: January 2027 (annual; content reviewed quarterly) Approver: Sarah Yoon, CISO Owner: Marcus Tan, Security Engineer (Detection & Response) Distribution: Information Security; Arctic Wolf MDR (under NDA)

1. Purpose

This Procedure specifies how MCT triages security alerts generated by Datadog Cloud SIEM, CrowdStrike Falcon, and Arctic Wolf MDR, from receipt through closure or escalation to incident response. It implements POL-001 §3.11 (logging and continuous monitoring) and §3.12 (incident response) and supports the detection use cases catalogued in PLAYBOOK-Detection-Use-Cases.

This Procedure is intentionally aligned with the SLAs that determine MCT's Time-to-Detect (TTD), Time-to-Investigate (TTI), and Time-to-Respond (TTR) KPIs.

2. Scope

This Procedure applies to all alerts originating from in-scope log sources (STD-LOG-01) regardless of consumer (MCT analyst, Arctic Wolf, automation). It does not cover incident response after declaration — that is governed by PLN-IR-01 and the PROC-IR-* family.

3. Roles and Responsibilities

5. Triage Workflow

5.1 Receipt

Alerts arrive via:

Datadog Cloud SIEM Signals -> Datadog Incidents (auto-paged for T1; queued for T2)

CrowdStrike Falcon detections -> Falcon console + Datadog mirror

Arctic Wolf alert escalation -> email + PagerDuty + Datadog incident note created by Arctic Wolf integration

Phishing reports from report@meridiancare.com mailbox routed via M365 reporting add-in

PagerDuty pages on-call for T1; T2 enters the queue with a 30-minute SLA timer.

5.2 Acknowledge

The on-call analyst (Iyer business hours; Arctic Wolf otherwise) acknowledges the alert in PagerDuty and opens / updates the corresponding Datadog incident.

5.3 Enrichment

Standard enrichment performed before assessment:

Identity context: Pull user profile from Okta (department, manager, last login geography, MFA factors).

Asset context: Pull host/container metadata from Wiz, CrowdStrike, EKS labels.

History: Datadog 30-day query for related signals on the same actor / asset / IP.

Threat-intel cross-reference: CrowdStrike Falcon Intelligence and AlienVault OTX feeds for IP/domain/hash IOCs.

Change context: Cross-reference ServiceNow change records and GitHub recent commits for legitimate explanation.

Known-good filters: Apply documented exclusions (synthetic monitoring sources, known build agents, scheduled workflows) before judging.

A Datadog Workbench Notebook template (triage-enrichment-template) is launched from each T1/T2 incident to capture this in a consistent format.

5.4 Assessment

The triage owner determines one of:

Benign — known cause. Closed with documented rationale. If recurring, file a tuning request (see §7).

Benign — unknown cause but verified safe. Closed with rationale; flag for monitoring.

Indeterminate — needs deeper investigation. Escalate per §6 if T1/T2 SLA at risk.

Suspected true positive. Escalate per §6 immediately.

5.5 Documentation

Every triage produces a Datadog incident note, regardless of disposition. The note includes:

Alert source and rule ID

Timeline (received, acknowledged, assessed, closed)

Enrichment artifacts (link to notebook)

Disposition and rationale

Tuning recommendation (if any)

Reviewer (second pair of eyes for T1)

Datadog incident notes are exported nightly to the cold archive (per STD-LOG-01) for evidence retention.

6. Escalation

6.1 Standing chain

For any alert that warrants escalation beyond triage:

Security Analyst (Iyer) -> Security Engineer D&R (Tan) -> CISO (Yoon)

Outside business hours, Arctic Wolf escalates directly to the on-call MCT engineer via PagerDuty, who then engages Tan and Yoon as needed.

6.2 When to escalate

Escalate immediately when any of the following is true:

Suspected compromise of a production identity, secret, or service account.

Confirmed unauthorized access to PHI or to a system that processes PHI.

Use case classified as Impact (IM-*) firing as anything other than a known scheduled action.

Falcon ransomware / encryption behavior detection.

Mass deletion or KMS key disable detected.

Alert that cannot be resolved within the T1 SLA window.

The CISO has authority under POL-001 §3.4 and PLN-IR-01 to declare an incident and assign a severity.

6.3 Escalation to incident

When triage transitions to incident, the Datadog incident is converted to an IR incident, and PROC-IR-* runbooks govern from that point. The triage incident note is preserved as the initial record.

7. Tuning Workflow

7.1 FP feedback loop

Every closure marked "Benign — known cause" emits a structured tuning record (Datadog tag tuning-candidate:true). Tan triages the tuning queue weekly and chooses one of:

Adjust the rule (threshold, filter, exclusion).

Add to the documented exclusion list (tracked in mct-detection-content).

Mark as accepted noise pending other work (e.g., legacy ETL decommission).

Promote to backlog for re-architecting the use case.

All rule changes follow the four-eyes review requirement and are tested in observation mode before redeployment per PLAYBOOK-Detection-Use-Cases §6.

8. Phishing Report Handling

Phishing reports from end users via the M365 reporting add-in (or report@meridiancare.com) are a special class:

Auto-acknowledged by the M365 / Defender pipeline.

Sample / cluster identified by Defender; suspicious clusters are queued for analyst review.

Iyer triages within 1 business hour of receipt during business hours; Arctic Wolf reviews overnight clusters.

Confirmed phishing -> remove from all mailboxes (M365 Threat Explorer remediation) -> targeted communication to recipients -> tuning record fed into PR.AT-* program.

Quarterly KPI: report rate (Q4 2025: 48%) and click rate on simulated phishing (Q4 2025: 2.4%).

9. Documentation, Audit, and Quality

All triage activity is captured in Datadog incidents and exported to the cold archive.

A weekly quality review by Tan samples 5 closed T1 and 5 closed T2 triages and verifies completeness. Findings are fed back to the analyst and into runbook revisions.

Quarterly metrics package to the Security Steering Committee covers volumes by tier, SLA attainment, FP rate, top-noisy rules, and tuning sprint progress.

Annual evidence package for SOC 2 Type II (Schellman) and HITRUST CSF r2 (Coalfire-validated): triage SLAs, FP rate, sample triages, escalation evidence.
