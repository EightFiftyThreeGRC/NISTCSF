1. Purpose

This document provides a logical view of the MCT production and corporate network architecture. It identifies trust zones, ingress and egress controls, identity boundaries, and the legacy on-premises footprint. It is referenced from POL-001 §3.9 and STD-ENC-01, and supports the HITRUST control narratives for 09.m, 09.n, and 01.j. It is intended as a logical reference, not a complete physical or VPC-level diagram; the latter are maintained in the infra-diagrams repository (auto-generated weekly from Terraform state and Wiz inventory).

3. Trust Zones

MCT recognizes five logical trust zones. The intent is that all traffic crossing a trust-zone boundary be authenticated, authorized, encrypted, and logged.

3.1 Honest acknowledgement — Z5 partial exception

Z5 is a known control gap acknowledged in POL-001 §3.9 and tracked as part of the FY26 strategy theme to decommission the legacy ETL cluster by Q4 2026. The cluster is excluded from full asset-inventory automation; patching is manual; and host-based detection coverage was extended to Wiz agent only in late 2025. Compensating controls are documented in CRA-LegacyETL-2025 (network segmentation, IPsec encryption, restricted egress, access via a single bastion path).

4. Ingress Controls

4.1 Customer-facing ingress (Patient, Clinician, Customer admin → MeridianCare)

DNS resolution to CloudFront edge (geo-fenced where customer contracts permit).

TLS 1.2+ termination at CloudFront. ECDSA certificates issued via AWS Certificate Manager with public ACM CA. HSTS enforced with 12-month max-age and preload.

AWS WAF v2 with the AWS Managed Rules Common Rule Set, the SQLi rule set, the Bot Control managed rule set (low-sensitivity for customer admin paths, higher-sensitivity for unauthenticated paths), and three custom MCT rules covering: (a) FHIR endpoint anomaly patterns, (b) high-volume read-only enumeration, (c) known-bad ASN list maintained by Arctic Wolf.

AWS Shield Standard. Shield Advanced is approved in the FY26 budget and pending Q2 2026 procurement.

ALB performs second TLS termination into the VPC; ALB is target-group restricted to ingress controllers in EKS; cross-zone load balancing enabled.

NGINX ingress controllers in EKS terminate the third TLS hop using cert-manager-issued certificates from a private ACM PCA. They route to namespaced Services within the cluster.

envoy sidecars enforce mTLS for east-west traffic between services where supported. Authorization is enforced via service-mesh policies sourced from SPIFFE identities.

4.2 EHR ingress (Direct Trust, FHIR R4, HL7 v2 from customers' EHRs)

Direct Trust messaging arrives at a dedicated ingestion service via an inbound DirectTrust HISP path (TLS + S/MIME). Path certificates are pinned.

FHIR R4 endpoints sit behind the same CloudFront / WAF / ALB chain as the customer-facing app, but on dedicated paths with stricter rate limits and per-customer mTLS where contractually agreed.

HL7 v2 feeds use AWS PrivateLink / VPN endpoints negotiated per customer; legacy v2 connections via the Raleigh on-prem cluster (Z5) are documented in §3.1.

All inbound clinical messages are persisted with sender/receiver metadata in Aurora; the message payload is parsed and stored encrypted, with the raw envelope retained for audit purposes.

4.3 Workforce ingress (MCT employees → corporate and production)

All workforce traffic to MCT systems traverses Cloudflare WARP. Device posture (Jamf for macOS, Intune for Windows) is checked at every connection.

Authentication via Okta. MFA is enforced; FIDO2/WebAuthn is mandatory for all production-access roles, security personnel, and engineers (full enforcement Q4 2025).

AWS access via IAM Identity Center; engineers and admins assume short-lived (1- to 4-hour) roles. No long-lived AWS access keys exist for human accounts; audited quarterly.

Production access additionally requires Okta-enforced step-up for sensitive role assumptions (e.g., prod-admin).

6. Logging and Monitoring

All ingress and egress events are logged. Specifically:

CloudFront and WAF logs delivered to S3 and ingested into Datadog.

VPC Flow Logs at the production VPC, retained 90 days hot and 7 years cold.

ALB access logs in S3.

Aurora pgAudit, Snowflake access history, S3 server access + CloudTrail data events.

Okta system log streamed to Datadog.

envoy mesh access logs in Datadog.

Datadog Cloud SIEM is the primary alerting surface; Arctic Wolf operates 24x7 escalation. Alert tuning remains an active area of investment (cyber risk register entry R3 — alert-tuning debt; not all detections are at production-grade fidelity).
